# 📋 الخطة التنفيذية الشاملة والمتكاملة: نظام تكلفتي SaaS المطور

توثق هذه الخطة الهيكل المعماري والبرمجي المتكامل لنظام **"تكلفتي SaaS"** لإدارة وحساب التكاليف الفعلية الشاملة للموظفين وتوزيعها ماليّاً وقانونيّاً وفقاً لمتطلبات المنشآت في المملكة العربية السعودية.

---

## 🏗️ 1. الهيكل المعماري للنظام (Architecture & SaaS Environment)

1. **عزل بيئة الشركات (SaaS Isolated Environment):**
   * يعتمد النظام على بنية **(Database-per-Tenant)**؛ حيث يتم إنشاء قاعدة بيانات منفصلة لكل شركة للتأمين التام للبيانات وعزلها.
   * **محلياً (Development):** يتم التخزين in ملفات JSON معزولة ديناميكياً لكل مستخدم (`backend/db/company_{id}.json`).
   * **سحابياً (Render / Production):** يتحول النظام تلقائياً للاتصال بقاعدة بيانات سحابية **PostgreSQL** مجانية ومستقلة، ويقوم السيرفر بإجراء مزامنة وتخزين ذكي (Write-Through Cache) لسرعة القراءة والكتابة الفائقة.
2. **الاستضافة الموحدة مجاناً (Single Instance Hosting):**
   * يتم دمج الواجهة الأمامية والخلفية في خدمة ويب واحدة (Express Backend + React Static Assets) لتوفير ساعات التشغيل المجانية على موقع Render ليعمل التطبيق 24 ساعة مجاناً مدى الحياة.

---

## 🏛️ 2. الهيكل التنظيمي المعتمد وقاعدة البيانات (Database Schema)

تتألف قاعدة البيانات من الجداول المترابطة التالية لتمثيل الهيكل القانوني والمالي الفعلي:

### أ. جدول الكيانات الكبرى (`entities`):
يمثل الكيانات القانونية الرئيسية للمجموعة (مثال: شركة ريال البركة للتجارة).
* `id` INTEGER PRIMARY KEY (المعرف الفريد)
* `name` TEXT (اسم الكيان الكبير)
* `unified_number` TEXT (الرقم الموحد للكيان الكبير 700)

### ب. جدول الفروع (`branches`):
يمثل الفروع التشغيلية التابعة للكيانات القانونية.
* `id` INTEGER PRIMARY KEY (المعرف الفريد للفرع)
* `name` TEXT (اسم الفرع)
* `cr_number` TEXT (رقم السجل التجاري الخاص بالفرع)
* `entity_id` INTEGER (معرف الكيان الكبير التابع له الفرع - علاقة ربط)

### ج. جدول المشاريع (`projects`):
* `id` INTEGER PRIMARY KEY (المعرف الفريد)
* `name` TEXT (اسم المشروع)
* `entity_id` INTEGER (الكيان الكبير التابع له المشروع، اختياري)
* `branch_id` INTEGER (الفرع التابع له المشروع، اختياري)

### د. جدول الموظفين (`employees`):
* `id` INTEGER PRIMARY KEY
* `employee_code` TEXT UNIQUE (الرقم الوظيفي)
* `name` TEXT (اسم الموظف)
* `nationality` TEXT (الجنسية)
* `gender` TEXT (النوع)
* `status` TEXT (حالة الموظف: على رأس العمل/مستبعد)
* `branch_id` INTEGER (الفرع القانوني المسجل عليه في التأمينات ونطاقات)
* `cost_branch_id` INTEGER (الفرع المالي الفعلي المتحمل للتكلفة والرواتب)
* `project_id` INTEGER (المشروع المنسب له الموظف)
* `saudi_type` TEXT (نوع التوطين للسعوديين فقط: `'working'` سعودي عامل، أو `'support'` دعم نشاط/سعودة فقط)
* `hire_date` TEXT (تاريخ مباشرة العمل)

### هـ. جدول الرواتب والبدلات (`salaries`):
* `employee_id` INTEGER PRIMARY KEY, `basic_salary` REAL, `housing_allowance` REAL, `transportation_allowance` REAL, `living_allowance` REAL, `other_allowances` REAL

### و. جدول الرسوم الإضافية للمقيمين (`resident_extra_costs`):
* `employee_id` INTEGER PRIMARY KEY, `medical_insurance_monthly` REAL, `health_certificate_monthly` REAL, `exit_reentry_monthly` REAL

## 📊 3. محرك الحساب المالي والمعادلات المعتمدة (Financial Cost Engine)

تُحسب التكاليف شهرياً لكل موظف وفقاً للمعادلات التالية المتطابقة مع نظام العمل ومحددات الشركة:

| البند المالي | معادلة الاحتساب في النظام |
| :--- | :--- |
| **التأمينات الاجتماعية (سعودي)** | `(Basic + Housing) * 22%` (أو النسبة المدخلة في الإعدادات) |
| **التأمينات الاجتماعية (مقيم)** | `(Basic + Housing) * 2%` (نسبة الأخطار المهنية) |
| **مخصص الإجازة شهرياً** | `(Basic / 30) * عدد أيام الإجازة السنوية (مثلاً 21) / 12` |
| **مخصص نهاية الخدمة شهرياً** | `(Basic / 2) / 12` (نصف راتب سنوي) |
| **مخصص تذاكر السفر شهرياً** | `قيمة التذكرة السنوية في الإعدادات / 12` |
| **مخصص رسوم الإقامة والجوازات** | `رسوم الإقامة السنوية في الإعدادات / 12` |
| **مخصص رخصة العمل (مكتب العمل)** | `رسوم مكتب العمل السنوية في الإعدادات / 12` |

### 🤖 حساب وتوزيع عبء السعودة تلقائياً 100% (Automated Saudization Burden):
1. **تكلفة السعودة الفعلية للكيان الكبير:** تُحسب أوتوماتيكياً بجمع إجمالي رواتب وتأمينات (22%) الموظفين السعوديين المسجلين **قانونياً** على فروع هذا الكيان.
2. **متوسط العبء للمقيم الواحد:** يُقسم إجمالي تكاليف التوطين لجميع الكيانات على إجمالي عدد المقيمين النشطين في المنشأة ككل.
3. **التطبيق المالي:** يضاف متوسط العبء شهرياً إلى التكلفة الإجمالية لكل مقيم بالتساوي.

## 🛡️ 5. تحسينات الجودة وتجربة المستخدم المعتمدة (UX & Validation)

1. **حماية الحذف التراكمي:** لا يُسمح بحذف الكيان القانوني إذا كان يحتوي على فروع، ولا يُسمح بحذف الفرع إذا كان يضم موظفين أو مشاريع نشطة لمنع تلف البيانات.
2. **الفلترة الديناميكية للمشاريع:** بمجرد اختيار الفرع المالي للموظف، تظهر فقط المشاريع التابعة له لمنع أخطاء التنسيب.
3. **نظام الفلترة الشامل:** فلاتر منسدلة أعلى لوحة التحكم وجدول الموظفين والـ Pivot للفرز حسب (الكيان الكبير، الفرع المالي/القانوني، أو المشروع).
4. **تقسيم لوحة التحكم (Compliance vs Operational Dashboard):**
   * **جدول الالتزام:** يعرض الكيانات والنسب ونقاط التوطين قانونياً.
   * **جدول التشغيل:** يعرض الفروع والمصاريف والرواتب المحملة تشغيلياً.
